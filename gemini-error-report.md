# IPO Master 프로젝트 문제 해결 및 기술 리포트

최근 `ipo-master` 프로젝트의 데이터 업데이트 및 알림 시스템에서 발생한 여러 기술적 이슈들의 원인, 분석, 그리고 최종 해결 방법을 정리한 문서입니다. 향후 유사한 문제 발생 시 가이드라인으로 활용하시기 바랍니다.

---

## 1. 데이터 파일(JSON) 오염 이슈

### 이슈 현황
- `ipo_list.json` 파일에 `<<<<<<<`, `=======`, `>>>>>>>`와 같은 Git 머지 충돌 마커가 포함되어 빌드 및 스크레이핑이 중단됨.

### 원인 분석
- **부적절한 워크플로우 순서**: GitHub Actions에서 최신 코드를 끌어오기(`git pull`) 전에 웹 스크래퍼가 실행되어 로컬 데이터가 변경됨.
- 이 상태로 `git pull`을 실행하면 자동 머지가 시도되나, 데이터 파일 구조 특성상 충돌이 잦아 충돌 마커가 파일 내에 그대로 삽입됨.

### 해결 방법
- **워크플로우 재정렬**: `data_update.yml`에서 **스크래퍼 실행 전에 `git pull --rebase`를 수행**하도록 순서를 변경.
- **클린업 스크립트 적용**: `clean-conflict-markers.js`를 작성하여 오염된 JSON에서 충돌 마커를 자동으로 제거하고 데이터 무결성을 회복함.

---

## 2. Next.js 빌드 및 정적 엑스포트 에러

### 이슈 현황
- `npm run build` 실행 시 `SyntaxError: Expected double-quoted property name` 에러 발생.

### 원인 분석
- 위에서 언급된 **JSON 파일 오염**으로 인해 `JSON.parse`가 실패함.
- Next.js의 `generateStaticParams`가 오염된 데이터를 읽어오지 못해 빌드 프로세스가 중단됨.

### 해결 방법
- JSON 데이터 클린업을 완료한 후 빌드가 정상적으로 완료됨을 확인. 데이터 무결성이 빌드 안정성의 핵심임을 확인.

---

## 3. 텔레그램 알림 시스템 불안정 (파워쉘 환경 문제)

### 이슈 현황
- 알림 메시지에 불필요한 파워쉘 객체 메타데이터가 포함되거나, 한글 깨짐, HTML 태그 깨짐(`\u003c` 등) 발생.

### 원인 분석
- **환경 의존적 동작**: 윈도우 자가 호스팅 러너(Self-hosted runner)의 파워쉘 버전에 따라 `Get-Content`가 raw 텍스트가 아닌 객체를 반환함.
- **이스케이프 이슈**: 파워쉘의 `ConvertTo-Json`이 HTML 특수 문자를 유니코드로 강제 변환하여 텔레그램 API에서 태그를 인식하지 못함.
- **Base64 패딩**: 쉘 명령어를 조합하는 과정에서 인코딩/디코딩 오류가 발생할 확률이 높음.

### 해결 방법
- **Node.js 통합 (Native axios)**: 모든 알림 로직(메시지 구성, HTML 이스케이프, API 호출)을 `generate-ipo-summary.ts`로 이전.
- **파워쉘 개입 최소화**: 워크플로우는 단순히 `npm run notify:telegram`을 실행만 하도록 하여 쉘 환경에 따른 변수를 완전히 제거.
- **HTML 이스케이프 적용**: 회사명 등 외부 데이터에 `escapeHtml`을 적용하여 텔레그램 파싱 에러(400 Bad Request) 방지.

## 4. 요약 스크립트 런타임 에러 (TypeError)

### 이슈 현황
- 스크립트 실행 중 `TypeError: Cannot read properties of undefined (reading 'split')` 발생.

### 원인 분석
- 스크래핑된 IPO 데이터 중 일부 항목의 `환불일`, `상장일` 등이 비어있는 경우, 해당 속성에 대해 `.split('.')`을 호출하려다 에러 발생.

### 해결 방법
- **Robust Helper 도입**: `isSameDate` 함수를 구현하여 `undefined` 여부를 먼저 체크하고 안전하게 날짜를 비교하도록 개선.

---

## 향후 관리 포인트 (Best Practices)
1. **표준 스택 우선**: 윈도우 쉘(PowerShell)보다는 런타임(Node.js) 내에서 로직을 처리하는 것이 OS 독립적인 안정성을 제공합니다.
2. **선행 Pull 필수**: 자동화 프로세스에서 데이터 파일을 수정하기 전에는 항상 `git pull`을 통해 최신 상태를 유지해야 합니다.
3. **방어적 프로그래밍**: 외부 데이터(스크래핑 등)를 다룰 때는 항상 값이 없을 수 있다는 가정(`null` 체크) 하에 로직을 작성해야 합니다.
